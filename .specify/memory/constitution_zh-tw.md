# Love Talk Card 專案憲章

## 核心原則

### 一、程式碼品質

所有原始碼必須（MUST）遵守乾淨程式碼標準，以可維護性、可讀性與
一致性為優先。

- 每個函式與模組必須（MUST）具有單一且明確的職責。
- 程式碼註解必須（MUST）使用繁體中文（ZH-TW）撰寫。
- 變數、函式與元件名稱必須（MUST）具描述性且能自我說明。除非為
  普遍認可的縮寫（如 `i18n`、`URL`、`ID`），否則禁止使用縮寫。
- 無用程式碼、未使用的匯入及被註解掉的程式碼區塊，在合併前
  必須（MUST）移除。
- 所有檔案必須（MUST）使用 LF 換行符。禁止使用 CRLF。
- 靜態分析與格式化工具必須（MUST）在 CI 中配置並強制執行。
  未通過檢查的程式碼不得（MUST NOT）合併。
- 複雜邏輯必須（MUST）包含繁體中文行內註解，說明「為什麼」
  而非「做什麼」。

**理由**：一款支援國際化的浪漫卡牌遊戲，需要任何貢獻者都能快速
理解的程式碼庫。嚴格的可讀性規範能縮短新人上手時間，並防止
面向使用者介面邏輯中的細微錯誤。

### 二、測試標準（不可妥協）

測試驅動開發（TDD）是所有功能開發的強制性開發方法。

- 必須（MUST）遵循紅-綠-重構循環：先撰寫失敗的測試，實作最少
  程式碼使其通過，再進行重構。
- 每個面向使用者的功能必須（MUST）有對應的單元測試，以及至少
  一個涵蓋主要使用者旅程的整合測試。
- 所有新程式碼的測試覆蓋率必須（MUST）達到或超過 80%。關鍵路徑
  （抽卡、狀態管理、i18n 切換）必須（MUST）達到 95% 覆蓋率。
- 測試必須（MUST）具確定性且相互隔離。任何測試不得依賴執行順序、
  網路可用性或外部狀態。
- 不穩定的測試必須（MUST）在發現後 24 小時內修復或隔離。被隔離
  的測試必須（MUST）建立追蹤 issue。
- 測試描述必須（MUST）清楚說明受測場景，使用以下模式：
  `[單元] [動作] [預期結果]`。

**理由**：卡牌遊戲依賴隨機化狀態（洗牌、私密模式切換）。TDD
確保這些有狀態互動在 UI 建構前就已驗證正確性，防止降低情侶
體驗的回歸問題。

### 三、使用者體驗一致性

所有 UI/UX 模式必須（MUST）在每個畫面與互動中保持一致、無障礙
且具響應式設計。

- 設計必須（MUST）採用行動優先（Mobile-First）方式，針對直式
  螢幕方向與單手操作進行最佳化。
- 互動元素（按鈕、卡片、切換開關）必須（MUST）滿足最小觸控
  目標 44×44 CSS 像素。
- 色彩對比必須（MUST）符合 WCAG 2.1 AA 標準（一般文字最低
  4.5:1，大型文字最低 3:1）。
- 主題轉場（依主題切換背景色）必須（MUST）使用平滑 CSS 過渡
  效果，持續時間介於 300ms 至 500ms 之間。
- 卡片翻轉動畫必須（MUST）在 600ms 內完成，且完成後不得
  （MUST NOT）阻擋使用者互動。
- 應用程式必須（MUST）支援國際化（i18n），至少支援繁體中文
  （ZH-TW）與英文（en）。預設顯示語言為繁體中文。
- 所有面向使用者的字串必須（MUST）外部化至翻譯檔案中。
  禁止在原始碼中硬編碼顯示字串。
- 所有日期/時間操作的預設時區必須（MUST）為 Asia/Taipei
  （UTC+8）。
- UI 必須（MUST）在初始資源快取後，透過 PWA 技術完整支援
  離線功能。

**理由**：這是一個以氛圍為核心的親密體驗。不一致的 UI、突兀的
動畫或在行動裝置上破碎的排版，會破壞產品所設計營造的浪漫氣氛。

### 四、效能要求

應用程式必須（MUST）快速載入且高效渲染，以維持沉浸式、不中斷
的體驗。

- 初始頁面載入（First Contentful Paint）必須（MUST）在 4G
  行動網路連線下於 1.5 秒內完成。
- 可互動時間（Time to Interactive）在中階裝置上必須（MUST）
  低於 3 秒。
- 初始套件總大小不得（MUST NOT）超過 200 KB（gzipped），不含
  已快取的 PWA 資源。
- 卡片翻轉動畫必須（MUST）維持 60 fps 且無掉幀。
- 圖片與資源檔案必須（MUST）經過最佳化：圖片使用 WebP/AVIF、
  圖示使用 SVG、字型使用 WOFF2。
- 初始視窗（viewport）不需要的資源必須（MUST）採用延遲載入
  （lazy loading）。
- Service Worker 必須（MUST）對靜態資源與卡牌資料實作
  快取優先（cache-first）策略，確保離線效能與線上使用一致。

**理由**：情侶們會在低連線環境（露營、飛行、偏遠小屋）中使用
此應用程式。緩慢的載入或卡頓的動畫會破壞沉浸感，降低持續使用
的意願。

## 專案限制與標準

以下限制適用於所有開發活動，並與核心原則一同強制執行。

- **程式碼註解語言**：所有程式碼註解必須（MUST）使用繁體中文
  （ZH-TW）。
- **提交訊息**：所有 Git 提交訊息必須（MUST）遵守
  Conventional Commits 格式。AI 產生的提交建議應先以繁體中文
  （ZH-TW）撰寫，再套入該格式。
- **換行符**：儲存庫中所有檔案必須（MUST）使用 LF（`\n`）
  換行符。必須（MUST）透過 `.gitattributes` 檔案強制執行。
- **國際化**：應用程式必須（MUST）至少支援繁體中文（ZH-TW）
  與英文（en）。第三語言支援（泰文、日文）為預計擴充。
- **時區**：所有日期、時間及排程操作的預設時區必須（MUST）為
  Asia/Taipei（UTC+8）。
- **外部化**：所有面向使用者的文字必須（MUST）外部化至翻譯
  資源檔案中。元件或服務程式碼中不得硬編碼任何顯示字串。
- **憲章鏡像同步**：英文版憲章（constitution.md）必須（MUST）
  與權威性的繁體中文版本
  （`.specify/memory/constitution_zh-tw.md`）保持同步。當兩者
  存在衝突時，繁體中文版本為標準來源。

## 開發工作流程

所有貢獻者必須（MUST）遵循此工作流程，以確保品質關卡一致達成。

1. **建立分支**：從 `main` 建立功能分支，遵循命名慣例
   `###-feature-name`。
2. **TDD 循環**：針對每個任務，先撰寫失敗的測試，實作程式碼使
   其通過，再進行重構。每次綠燈階段後提交。
3. **靜態分析與格式化**：推送前在本地端執行靜態分析與格式化
   檢查。CI 必須（MUST）將這些檢查作為合併閘門。
4. **程式碼審查**：每個 Pull Request 必須（MUST）在合併前經過
   審查。審查者必須（MUST）驗證是否符合所有四項核心原則。
5. **憲章合規**：審查者必須（MUST）確認：
   - 程式碼註解使用繁體中文。
    - 提交訊息遵守 Conventional Commits 格式。
    - AI 產生的提交建議以繁體中文（ZH-TW）撰寫。
   - 所有新的面向使用者字串已外部化。
   - 新元件符合無障礙與效能門檻。
6. **持續整合**：CI 管線必須（MUST）在每次推送時執行完整測試
   套件、靜態分析檢查及建置驗證。CI 失敗必須（MUST）阻止合併。
7. **提交粒度**：每次提交必須（MUST）代表一個單一邏輯變更。
   允許壓縮提交（squash）進行清理，但必須（MUST）保留有意義
   的歷程。

## 治理

本憲章為 Love Talk Card 專案的最高治理文件。它優先於所有其他
慣例、指南或非正式實踐。

- **修訂流程**：對本憲章的任何變更必須（MUST）透過 Pull Request
  提出，並附上明確理由。變更必須（MUST）在合併前經過審查與批准。
- **版本策略**：本憲章遵循語意化版本（Semantic Versioning）：
  - MAJOR：移除或不相容地重新定義核心原則。
  - MINOR：新增原則、章節，或對現有指引進行重大擴充。
  - PATCH：澄清說明、錯字修正或非語意性的改善。
- **合規審查**：所有 Pull Request 與程式碼審查必須（MUST）驗證
  是否符合現行憲章版本。違規事項必須（MUST）在合併前解決。
- **雙語維護**：`constitution.md`（英文）與
  `constitution_zh-tw.md`（繁體中文）必須（MUST）在同一次提交
  中更新。繁體中文版本為權威版本。
- **指引參考**：使用 `AGENTS.md` 作為 AI 輔助開發工作流程的
  執行時期開發指引。
